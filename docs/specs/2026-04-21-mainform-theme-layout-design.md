# MainForm 主题布局修复设计

日期：2026-04-21

> 说明：这份设计记录的是上一轮“主题布局修复”的范围，已被 [2026-04-21-mainform-theme-startup-parity-design.md](/mnt/d/dev/weiqi/readboard/docs/superpowers/specs/2026-04-21-mainform-theme-startup-parity-design.md) 补充并作为当前执行入口。后续如果涉及“启动主题不同导致窗口尺寸和布局分叉”的修复，以新文档为准。

## 背景

在几轮针对不同分辨率和高 DPI 的主窗体兼容修复之后，`MainForm` 的主题布局出现了三类回归：

1. 新主题在主窗体头部区域出现了明显的大块空白。具体表现为：右侧工具列保留在顶部，但 `棋盘规格` 的起始位置被错误地锚定到了 `检查更新` 按钮的底部，导致 `平台类型` 与 `棋盘规格` 之间出现不合理的大间隙。
2. 默认主题与新主题已经不再共享同一套布局结果。原始设计预期是主题之间只切换视觉样式，但当前实现中布局测量直接依赖当前主题字体和控件即时测宽，造成按钮布局、分组位置和窗口大小出现明显分叉。
3. 主题命名需要调整。当前界面文案中的 `修复版主题` 需要统一改为 `新版主题`，但不改变已有配置值和兼容逻辑。

用户另外确认了一个布局细节：`同步与自动落子` 分组中的 `最大计算量(选填)` 与 `首选计算量(选填)` 需要严格上下对齐。

## 目标

本次修复的目标如下：

- 以当前用户提供的主窗口截图作为唯一布局基准。
- 让默认主题与新版主题共用同一套主窗口布局骨架。
- 消除 `平台类型` 与 `棋盘规格` 之间的不合理空白。
- 让 `同步与自动落子` 分组内的两行标签与输入框形成严格的共享列对齐。
- 保持不同分辨率、不同 DPI 缩放和窄宽度场景下的可用性与稳定性。
- 将界面显示文案中的 `修复版主题` 统一更名为 `新版主题`。

## 非目标

- 不重做整个 WinForms 主窗体结构。
- 不改动配置存储格式，不变更 `UiThemeMode` 的取值语义。
- 不借机重构其他窗体或打包流程。
- 不改变与 `LizzieYzy-Next` 的集成方式。

## 用户体验与视觉方向

本界面采用以下确认后的设计方向：

- 目标气质：专业、稳定、工具感强。
- 视觉基准：尽量贴近用户提供的当前截图，不进行激进现代化改造。
- 主题关系：默认主题和新版主题在布局上完全一致，只允许颜色、边框、按钮视觉和字体表现不同。

## 方案比较

### 方案 1：补丁式修复

直接修正头部起始高度、同步区局部对齐和主题命名。

优点：

- 改动小。
- 能快速消除最明显的视觉问题。

缺点：

- 默认主题与新主题布局分叉的根因仍然存在。
- 后续只要主题字体或按钮文案变化，仍可能重新出现不同布局结果。

### 方案 2：共享布局度量，主题只改样式

为 `MainForm` 建立一套与主题无关的共享布局度量和排版规则，两个主题仅共享这套规则并分别应用自己的视觉样式。

优点：

- 同时解决头部空白、主题分叉和同步区错位问题。
- 最符合“原本只有样式不同，布局全都一样”的产品定义。
- 更适合继续维护 DPI/分辨率兼容。

缺点：

- 比补丁式修复多改一部分布局计算代码。

### 方案 3：改为更强约束的容器化布局

使用新的容器布局体系重做主窗体头部与同步区。

优点：

- 结构更强约束。

缺点：

- 对现有 WinForms 工具界面的扰动过大。
- 容易引入新的 DPI 和兼容性回归。

## 选定方案

选择方案 2：共享布局度量，主题只改样式。

原因：

- 当前两个主要问题都来自同一个根因：布局度量与主题样式耦合。
- 单点打补丁只能缓解现象，不能恢复“两个主题共用同一布局骨架”的定义。
- 共享布局度量能够在不大改主窗体结构的前提下收敛风险，并保留现有 DPI 兼容框架。

## 设计细节

### 1. 主窗口布局骨架

`MainForm` 保持一套共享的纵向分区顺序：

1. `平台类型`
2. `棋盘规格`
3. `同步与自动落子`
4. 操作按钮区

在宽窗口场景下：

- 左侧为 `平台类型` 分组。
- 右侧为工具按钮列，保留 `参数设置 / 帮助 / 主题 / 6.5目规则设置方法 / 检查更新` 的现有结构。
- `棋盘规格` 的起始 Y 坐标只依赖 `平台类型` 分组底部，不再依赖右侧工具列中最底部按钮的底边。
- `同步与自动落子` 的起始 Y 坐标只依赖 `棋盘规格` 分组底部。

在窄窗口或高 DPI 放大导致横向空间不足时：

- 仍允许现有自适应换行逻辑生效。
- 但纵向锚点关系保持不变，即每个分组只跟随前一个分组，而不是跟随右侧工具列的最底部。

### 2. 共享布局度量

`MainForm` 的布局计算将从“按当前主题即时测宽后直接排版”调整为“共享逻辑度量 + 必要安全宽度”。

共享布局度量至少包括：

- 主窗体默认逻辑宽高
- 分组间距
- 按钮高度
- 工具列宽度策略
- `棋盘规格` 的输入框宽度与间距
- `同步与自动落子` 的标签列和输入列宽度

主题切换后：

- 布局仍使用同一套逻辑度量。
- 主题只负责应用视觉样式。
- 不允许因为主题不同而改变主窗口布局结果。

### 3. 同步区对齐规则

`同步与自动落子` 分组保留两行结构，但两行改为共享列模型：

- 第 1 行：`双向同步` / `执黑` / `引擎自动落子条件` / 输入框 / `最大计算量(选填)` / 输入框
- 第 2 行：`自动落子` / `执白` / `每手用时` / 输入框 / `首选计算量(选填)` / 输入框

共享规则如下：

- 第 3 列标签宽度共享。
- 第 4 列输入框宽度共享。
- 第 5 列标签宽度共享。
- 第 6 列输入框宽度共享。

结果要求：

- `最大计算量(选填)` 和 `首选计算量(选填)` 左边界、宽度、输入框左边界和输入框宽度一致。
- `引擎自动落子条件` 和 `每手用时` 也按相同规则对齐。
- 在宽度不足时，以列块为单位整体换行，而不是让单个标签或输入框随机漂移。

### 4. 主题命名

仅修改界面显示文案：

- 中文：`修复版主题` -> `新版主题`
- 其他语言资源同步更新为对应的新名称

兼容边界：

- 不修改 `UiThemeMode` 的数值。
- 不修改已有配置读写格式。
- 不影响旧配置文件继续读取到当前主题模式。

### 5. DPI 与分辨率兼容

本次修复继续基于现有的 `ScaleValue(...)`、`ScaleSize(...)`、工作区约束和 `AutoScroll` 机制实现，不额外引入静态像素布局。

要求如下：

- 所有新增或调整的间距、列宽和最小宽度都以逻辑尺寸参与缩放。
- 主窗体宽高仍受工作区限制，避免在较小分辨率或高缩放下超出可用区域。
- 当内容高度超过可用空间时，继续依赖滚动而不是裁切控件。
- 主题切换、`检查更新` 文案变化为 `检查中` 时，不应造成新的错位或截断。

## 代码边界

本次只涉及以下范围：

- `readboard/Form1.cs` 中的主窗体共享布局逻辑
- `readboard/Program.cs` 和各语言资源文件中的主题显示文案
- 与主窗体布局回归相关的测试

不涉及：

- 其他窗体主题布局
- 配置模型与持久化格式
- 发布脚本、CI 和宿主集成契约

## 测试与验证

### 自动验证

应补充或更新以下类型的回归测试：

- 主窗体高 DPI 布局源码回归测试，锁住新的共享布局代码路径和关键约束。
- 主题显示名称资源测试，锁住 `新版主题` 文案。

### 人工验证

至少覆盖以下场景：

- 默认主题与新版主题切换后，主窗体分组位置一致。
- `棋盘规格` 紧接 `平台类型` 下方。
- `同步与自动落子` 紧接 `棋盘规格` 下方。
- `最大计算量(选填)` 与 `首选计算量(选填)` 严格对齐。
- `100% / 125% / 150% / 200%` DPI 下无异常空白、错位或裁切。
- 窄宽度下允许换行但不破坏共享对齐关系。
- `检查更新` 按钮切换到 `检查中` 文案时仍保持稳定。

## 风险与控制

主要风险：

- 某些宽度阈值如果仍然依赖当前 `PreferredSize`，主题切换后可能继续出现分叉。
- 同步区在极窄宽度下如果换行规则不一致，可能再次出现局部错位。

控制方式：

- 将关键列宽和最小宽度统一收拢到共享布局度量中。
- 只允许在一处调整阈值，避免给不同主题分别打补丁。
- 保留现有 DPI 缩放与滚动机制，不引入新的隐藏 fallback 或例外路径。
